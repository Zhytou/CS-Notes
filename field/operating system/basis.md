## Linux

操作系统的本质就是对硬件的抽象。它将硬件的复杂性隐藏起来，为应用程序提供统一的接口。以Linux为例，它对硬件做了三层抽象：

- 进程抽象：把 CPU 虚拟化 → 让每个程序以为独占 CPU
- 虚拟内存抽象：把物理内存虚拟化 → 隔离、保护、扩展
- 文件抽象：把一切设备、资源的访问都简化成read/write/ioctl函数调用

### Linux内核与系统调用

Linux内核是Linux系统的核心，是操作系统与硬件直接交互的唯一入口。它由Linus Torvalds主导开发，几乎全部用C语言（少量汇编）实现，运行在内核态（最高特权级），负责管理所有硬件资源和核心功能。

**系统调用**：

为了使用户进程能够使用操作系统更底层的功能，内核会暴露出一些标准化接口，这些接口就是系统调用，是用户进程访问内核资源的唯一合法途径。
为了保证操作系统的稳定和安全，内核依据CPU提供的特权级别，建立了两种特权状态，即内核态和用户态：

- 当CPU处于用户态时，只能访问用户空间，权限受限，无法直接操作硬件和内核空间，避免用户进程误操作破坏系统；

- 当CPU处于内核态时，拥有最高权限，能同时访问用户空间和内核空间，可直接操作所有硬件资源、执行内核核心操作。

因此，当用户进程发起一个系统调用时，CPU会从用户态切换至内核态，执行相应的内核操作完成底层功能调用，操作结束后，再切换回用户态，将结果返回给用户进程，形成完整的调用闭环。

### 标准C库与GNU工具链

除了Linux内核外，现代Linux系统中往往还包含大量用户态组件，如标准C库、GNU工具链等。

**标准C库（libc）**：

其中，标准C库（libc）和GNU工具链是用户态最核心的组件之一。它们为应用程序提供了基本的函数和工具，如文件操作、内存分配、字符串处理等，极大降低了应用程序的开发难度。

标准C库本质上是对系统调用的封装和扩展，比如：应用程序中常用的fopen、fread、fwrite函数，其底层就是对内核提供的open、read、write等系统调用的封装。

**GNU工具链**：

GNU工具链是一组开源工具的集合，是Linux用户态的核心操作工具，与标准C库协同工作，构成用户态的基础环境，主要包括：

- Shell（命令解释器）：如bash、zsh，是用户与系统交互的界面，接收用户输入的命令（如ls、cd、mkdir），解析并调用对应工具或系统调用执行；

- 基础命令工具：如ls（查看目录）、cp（复制文件）、rm（删除文件）、ps（查看进程）等，用于日常系统操作，底层依赖标准C库和系统调用；

- 编译工具：如gcc（C语言编译器）、g++（C++编译器）、ld（链接器），用于将源代码编译成可执行程序，编译过程中会调用标准C库的函数接口；

- 调试工具：如gdb（程序调试器），用于排查程序运行错误，辅助开发者定位问题。

注意：Linux系统常被称为“GNU/Linux”，正是因为其用户态工具（包括GNU工具链、部分标准C库）几乎都来自GNU项目，内核来自Linux项目，两者协同工作，才构成了完整、可使用的Linux操作系统。

### 固件 Firmware

固件是预先烧录在主板ROM（Read-Only Memory，只读存储器）中的底层软件,负责初始化硬件并为操作系统加载提供基础运行环境。它是计算机通电后CPU执行的“第一行代码”，不依赖于任何磁盘设备而存在。常见的计算机固件包括BIOS（Basic Input/Output System，基本输入输出系统）以及UEFI（Unified Extensible Firmware Interface，统一可扩展固件接口）。

以BIOS为例，在计算机通电后，它会按照如下流程执行：

- 首先，执行加电自检，逐一检测CPU、内存、硬盘、显卡等核心硬件的完整性和可用性，若检测到硬件故障，会通过蜂鸣音或简单提示反馈故障信息；
- 随后，当自检通过后，初始化键盘、鼠标、磁盘等基础硬件，建立最基本的硬件通信链路；
- 最后，B照预设的启动顺序，读取启动设备的指定区域，加载引导程序，并将系统控制权移交出去。

### 引导程序 Bootloader

引导程序运行于固件之后、操作系统内核之前，是连接固件与内核的关键环节，其核心任务是精准定位Linux内核镜像在磁盘中的位置，将内核加载到内存中，同时传递内核启动所需的参数，最终将系统控制权完整移交给内核，完成自身使命。

常见的引导加载程序种类多样，适配不同的操作系统和启动场景，包括GNU GRUB（Linux系统中最主流的引导程序）、rEFInd（支持多系统引导，界面更友好）、Windows系统的BOOTMGR、Syslinux（轻量型引导程序，常用于U盘启动）以及旧版Windows系统使用的NTLDR。需要特别注意的是，这些引导程序本身并非操作系统，不具备资源管理、进程调度等核心功能，但具备解析磁盘文件系统、定位内核镜像、实现程序跳转执行的核心能力，是内核能够正常启动的重要前提。

### 内核初始化

当引导程序将内核加载到内存并移交控制权时，系统仍处于“几乎空白”的状态：没有虚拟内存、没有进程、没有栈、没有文件系统，甚至CPU还处于实模式（16位，只能访问1MB内存），而Linux内核几乎全是C语言编写的，C语言的运行需要栈和内存空间——这就是核心疑问：内核如何在这种“一无所有”的环境中启动？

答案：内核启动的第一步，用汇编代码搭建“能运行C语言的最小环境”，再切换到C语言，逐步完成初始化，整个过程分为3个阶段：

**汇编初始化（内核入口，搭建C语言运行环境）**：

Linux内核的启动入口是汇编代码（如x86架构下的arch/x86/kernel/head_64.S），这部分代码是内核启动的“基石”，仅做最核心的几件事，目的是让CPU和内存达到“能运行C语言”的状态：

- 切换CPU模式：将CPU从“实模式”（16位，内存限制1MB）切换到“保护模式”（32位）或“长模式”（64位），突破内存限制，支持访问更大的物理内存；

- 开启分页机制（虚拟内存）：初始化页表（将虚拟地址映射到物理地址），开启虚拟内存——这是核心步骤，虚拟内存并非C语言创建，而是汇编代码先开启，为后续C语言运行提供内存隔离和保护；

- 初始化栈（stack）：C语言的函数调用、局部变量都依赖栈，汇编代码会在物理内存中分配一块区域，设置栈指针（sp），建立起C语言运行所需的栈环境；

- 初始化内核全局变量：清零内核的BSS段（未初始化的全局变量），确保变量初始值正确；

- 跳转到C语言入口：完成以上步骤后，汇编代码通过跳转指令，进入内核的第一个C语言函数——start_kernel()，自此，内核从汇编阶段进入C语言阶段。

关键总结：C语言的运行不需要“完整的系统环境”，只需要“栈 + 可执行代码 + 内存（虚拟内存已开启）”，而这三者都是汇编代码在启动初期搭建好的，并非内核启动后才创建。

**C语言初始化（start_kernel()，构建完整内核环境）**：

start_kernel()是内核所有初始化工作的入口，相当于内核的“main函数”，此时虚拟内存已开启、栈已存在，C语言可以正常运行，该函数会逐步初始化内核的所有核心模块，从“最小环境”扩展为“可运行的内核环境”：

- 内存管理初始化：初始化页分配器、slab分配器等，建立完整的内存管理机制，让内核可以正常分配和使用物理内存、虚拟内存；

- 进程管理初始化：创建Linux系统的第一个进程——进程0（idle task，空闲进程），这是所有进程的父进程，后续所有进程都由它衍生而来；初始化进程调度器，实现进程的调度和切换；

- 文件系统初始化：挂载initramfs临时文件系统，加载所需的驱动程序；之后挂载根文件系统（/），建立完整的文件系统环境；

- 设备驱动初始化：初始化各种硬件设备的驱动（如磁盘、网卡、键盘），让内核可以正常控制硬件；

- 创建init进程：init进程是用户态的第一个进程，负责启动后续的用户态服务和应用程序，是用户态的“根进程”。

**用户态启动（完成系统启动）**：

当内核完成所有初始化，创建init进程后，会将进程1切换到用户态，由进程1启动：

- 加载标准C库（libc）：为后续用户态程序提供系统调用封装；

- 启动系统服务：如ssh、crond等后台服务；

- 启动桌面环境或Shell：若为图形化系统，启动GNOME/KDE；若为命令行系统，启动bash等Shell；

- 等待用户操作：系统启动完成，进入可交互状态，用户可以通过命令或图形界面操作系统。
